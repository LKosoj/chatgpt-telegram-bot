# Аудит проекта - 2026-05-13

Объединённая версия аудита. В списке оставлены только подтверждённые
архитектурные, логические, конкурентные, эксплуатационные и тестовые проблемы.

Проверка выполнялась по коду, без вызова внешних сервисов. Файл содержит
ссылки на актуальные строки исходников на момент аудита.

## Шкала

- HIGH - пользовательский сценарий ломается, данные сессий могут потеряться или
  состояние становится недетерминированным.
- MEDIUM - дефект проявляется при конкретной конфигурации, нагрузке или типе
  входных данных.
- LOW - тестовый пробел, документационный drift или проблема сопровождения.

## HIGH

### 1. Валидный JSON не-объект в tool-call аргументах валит весь handler

В `bot/openai_tool_handler.py:496` аргументы tool-call декодируются через
`json.loads(arguments)`, после чего код сразу мутирует результат как dict:
`args['chat_id'] = ...` на `bot/openai_tool_handler.py:498` и
`bot/openai_tool_handler.py:503`. Обрабатывается только `json.JSONDecodeError`
(`bot/openai_tool_handler.py:516`), а `TypeError` для `[]`, `"x"`, `1` или
`null` уходит во внешний `except` и пробрасывается дальше
(`bot/openai_tool_handler.py:603`).

Почему это баг: модель может вернуть синтаксически валидный JSON не-объект.
Один такой tool-call срывает весь ответ вместо того, чтобы добавить
recoverable tool error в историю.

Что сделать: после `json.loads` проверять `isinstance(args, dict)`;
не-dict трактовать как invalid arguments и добавлять в `errors`.

### 2. Инвариант "одна активная сессия на пользователя" не закреплён

Схема `conversation_context` имеет primary key только по `(user_id, session_id)`
(`bot/database.py:156-170`), но не запрещает несколько строк с
`is_active = 1`. `save_conversation_context()` при insert новой строки явно
ставит `is_active = 1` (`bot/database.py:407-414`) и не деактивирует остальные.
`get_active_session_id()` возвращает первую найденную активную запись без
`ORDER BY` (`bot/database.py:353-357`). При этом `switch_active_session()` явно
демонстрирует ожидаемый инвариант: сначала деактивирует все сессии пользователя,
потом активирует одну (`bot/database.py:896-908`).

Почему это баг: чтение активной сессии становится недетерминированным. Разные
ветки могут продолжить разные контексты одного пользователя.

Что сделать: закрепить инвариант на уровне кода и БД. Минимум - перед insert
новой активной сессии деактивировать остальные в той же транзакции. Лучше -
частичный unique index для `user_id WHERE is_active = 1`, если миграция
допустима.

### 3. `create_session()` может удалить текущую активную сессию до копирования настроек

Докстринг обещает создание новой сессии с сохранением режима из активной
(`bot/database.py:720-729`). Но сначала выполняется prune старых сессий
(`bot/database.py:732-735`), а только потом читается активная сессия
(`bot/database.py:737-740`). Выбор старых сессий не исключает активную:
`get_oldest_session_ids_for_limit()` сортирует все сессии пользователя по
`created_at` (`bot/database.py:643-650`).

Почему это баг: если активная сессия самая старая и достигнут лимит, она может
быть удалена до чтения режима, parse mode, temperature и system prompt.
Новая сессия тогда создаётся как пустая fallback-сессия
(`bot/database.py:748-780`).

Что сделать: сначала считать активную сессию и её настройки, затем prune; либо
исключать активную из кандидатов на удаление.

### 4. Создание и переключение сессий не атомарны при concurrent updates

Telegram application включён с `concurrent_updates(True)`
(`bot/telegram_bot.py:4124-4128`). Текстовые сообщения защищены
conversation-lock (`bot/telegram_bot.py:2519-2523`), но callback-ветки сессий
мутируют состояние напрямую: создание (`bot/telegram_bot.py:3979-3988`),
переключение (`bot/telegram_bot.py:4004-4017`), удаление
(`bot/telegram_bot.py:4019-4030`) и смена режима
(`bot/telegram_bot.py:1728-1765`). Внутри `create_session()` prune, чтение
активной сессии и insert выполняются отдельными DB-операциями
(`bot/database.py:732`, `bot/database.py:738`, `bot/database.py:795`).

Почему это баг: два callback-запроса или callback плюс входящее сообщение могут
одновременно создавать/переключать сессии. Возможны превышение `max_sessions`,
две активные сессии и рассинхрон `self.openai.conversations` с БД.

Что сделать: использовать тот же conversation-lock для session/mode callbacks и
свести prune/deactivate/insert к одной транзакции.

### 5. `ask_sync()` блокирует async-поток и зависит от пустого `OPENAI_BASE_URL`

В async path auto-mode напрямую вызывает синхронный `ask_sync()`
(`bot/openai_helper.py:749-758`). `ask_sync()` собирает URL из
`self.config['openai_base']` (`bot/openai_helper.py:2250`) и делает
`requests.post(...)` без timeout (`bot/openai_helper.py:2263-2268`). При этом
конфиг по умолчанию задаёт `openai_base` пустой строкой
(`bot/__main__.py:77-79`).

Почему это баг: event loop Telegram-бота может зависнуть на сетевом вызове.
Кроме того, при пустом `OPENAI_BASE_URL` auto-mode и генерация названия сессии
могут падать не так, как основной async OpenAI-клиент.

Что сделать: заменить `ask_sync()` на async-вызов через общий OpenAI client или
вынести синхронный HTTP в executor с timeout и явной проверкой base URL.

### 6. Image document ломает vision handler

Handler зарегистрирован для `filters.PHOTO | filters.Document.IMAGE`, и ветка
document сохраняет `update.message.document.file_id`
(`bot/telegram_bot.py:2167-2170`). Но дальше код берёт
`update.message.effective_attachment[-1]` (`bot/telegram_bot.py:2188`), что
работает для списка фото, но не для document attachment.

Почему это баг: отправка изображения как Telegram document - поддержанный
фильтром вход, но этот путь может упасть до vision-вызова.

Что сделать: выбирать attachment по типу: `photo[-1]` для фото и
`document` для image document.

## MEDIUM

### 7. Выбор `big_context`/big model теряется после tool-call re-entry

Первичный запрос переключает `model_to_use` на `big_model_to_use`, если история
слишком длинная или `big_context=True` (`bot/openai_helper.py:837-840`). Но
`max_tokens` был рассчитан до переключения (`bot/openai_helper.py:797-800`) и
передаётся дальше без пересчёта (`bot/openai_helper.py:851-856`). После tool-call
модель заново берётся через `get_current_model(user_id)`
(`bot/openai_tool_handler.py:565-586`), а retry empty response делает то же самое
(`bot/openai_helper.py:1017-1069`, `bot/openai_helper.py:1071-1082`).

Почему это баг: длинный запрос может стартовать на big model, но финальный ответ
после tools уйдёт на обычную модель или с лимитом токенов от старой модели.

Что сделать: хранить выбранную модель и max_tokens в request context и
передавать их через весь tool-call/retry pipeline.

### 8. Streaming-путь молча сохраняет пустой assistant response

Streaming собирает `answer`, делает `answer.strip()`, затем сразу сохраняет его
в историю (`bot/openai_helper.py:630-646`) и отдаёт наружу
(`bot/openai_helper.py:658`). В non-streaming path для пустого ответа есть
отдельная проверка через `_required_choice_message_text()` (`bot/openai_helper.py:94-105`).

Почему это баг: пустой streaming response выглядит для пользователя как
отсутствие ответа, но в БД появляется пустой assistant turn.

Что сделать: после strip проверять пустую строку и запускать тот же retry/error
path, что у non-streaming.

### 9. Vision follow-up не сохраняет изображение как контекст

При `enable_vision_follow_up_questions` код ставит
`self.conversations_vision[chat_id] = True` (`bot/openai_helper.py:1357-1359`),
но в историю сохраняет только текстовые части через `_vision_history_content()`
(`bot/openai_helper.py:1355`, `bot/openai_helper.py:1432-1439`). Обычный chat
path затем использует текстовую модель (`bot/openai_helper.py:851-856`), а
tools отключаются, пока `conversations_vision[chat_id]` true
(`bot/openai_helper.py:883-889`).

Почему это баг: follow-up вопрос по изображению не содержит само изображение в
контексте, но при этом меняет поведение обычного чата и отключает tools.

Что сделать: либо хранить image content/ссылку и отправлять follow-up в vision
path, либо не включать vision-state после одиночного image response.

### 10. Pre-delete hook получает snapshot, но удаление происходит по устаревшему списку

`_dispatch_and_delete_oldest_sessions_for_limit()` сначала получает список
session ids (`bot/telegram_bot.py:536-538`), затем по каждому awaits hook
(`bot/telegram_bot.py:540-541`) и только после этого удаляет исходный список
(`bot/telegram_bot.py:542-543`). Snapshot для hook читается отдельно из БД
(`bot/telegram_bot.py:505-507`). `delete_sessions_by_ids()` возвращает `True`
без проверки количества реально удалённых строк (`bot/database.py:655-666`).

Почему это баг: при параллельных callback-операциях hook может получить snapshot
сессии, которая уже изменилась или была удалена, а итоговое удаление не замечает
stale ids.

Что сделать: брать lock/транзакцию вокруг select-hook-delete либо после hooks
перепроверять кандидатов и rowcount.

### 11. Preview сессии может превысить лимит Telegram и падает вне локального try

Preview добавляет до 200 символов из каждого сообщения
(`bot/telegram_bot.py:3939-3948`) и отправляет весь текст одним
`query.edit_message_text()` (`bot/telegram_bot.py:3966-3969`). Ближайший `try`
начинается только после этой ветки (`bot/telegram_bot.py:3972`).

Почему это баг: длинная сессия может превысить лимит Telegram-сообщения, а
ошибка уйдёт в глобальный handler вместо локальной деградации интерфейса.

Что сделать: ограничивать общий размер preview и поместить отправку в локальный
try с коротким fallback-сообщением.

### 12. `/stats` блокирует event loop на внешнем balance call

Команда stats в async handler добавляет баланс VseGPT для админа
(`bot/telegram_bot.py:835-840`). `get_credits()` внутри использует
синхронный `requests.get()` без timeout (`bot/telegram_bot.py:853-860`).

Почему это баг: один `/stats` может задержать обработку других update в том же
event loop.

Что сделать: заменить на async HTTP client с timeout или выполнять в executor.

### 13. Framework-injected аргументы валидируются как model-provided аргументы

`openai_tool_handler` добавляет `chat_id`, `user_id`, `message_id` к аргументам
до вызова `PluginManager.call_function()` (`bot/openai_tool_handler.py:496-504`).
Затем `PluginManager` валидирует уже расширенный объект по JSON schema
(`bot/plugin_manager.py:295-304`). Валидатор возвращает ошибку на
`additionalProperties` (`bot/validation.py:71-72`).

Почему это баг: разработчик plugin не может использовать строгую schema с
`additionalProperties: false`, если не добавит framework-поля в публичный spec.

Что сделать: валидировать только model-provided args, а framework context
передавать отдельно через `request_context` или после validation.

### 14. Поиск владельца tool-функции не изолирован от сломанного плагина

`PluginManager.call_function()` до основного `try` ищет plugin по имени функции
(`bot/plugin_manager.py:280`). Поиск сканирует все plugins и вызывает
`get_plugin()`/`get_spec()` без per-plugin isolation
(`bot/plugin_manager.py:406-414`, `bot/plugin_manager.py:463-470`).

Почему это баг: исключение в чужом plugin constructor, initialize или get_spec
может сорвать вызов валидной функции другого плагина.

Что сделать: изолировать lookup per plugin: логировать исключение и продолжать
сканирование, а для зарегистрированных specs держать готовый индекс
`function_name -> plugin`.

### 15. Startup background tasks не изолирует падение `get_plugin()`

`start_background_tasks()` обходит plugins и вызывает `self.get_plugin(plugin_name)`
до `try`, который защищает только `plugin.get_background_tasks()`
(`bot/plugin_manager.py:943-953`). `get_plugin()` может создать instance и вызвать
initialize (`bot/plugin_manager.py:497-515`).

Почему это баг: один plugin с ошибкой construct/initialize может прервать запуск
background tasks остальных plugins.

Что сделать: обернуть `get_plugin()` в тот же isolation block и продолжать
следующий plugin после логирования.

### 16. `.env.example` включает plugins, но модель по умолчанию отключает tools

В `.env.example` указаны `OPENAI_MODEL=gpt-4.1-mini` и `PLUGINS=...`
(`.env.example:28`, `.env.example:65`). При этом `enable_functions` по умолчанию
берётся из `are_functions_available(model)` (`bot/__main__.py:94-97`), а эта
функция разрешает tools только для `LLMGATEWAY_CHAT_MODELS`
(`bot/openai_helper.py:149-154`, `bot/model_constants.py:3-11`).

Почему это баг: пользователь копирует sample env, видит список plugins, но
tool-calls по умолчанию не включаются.

Что сделать: синхронизировать sample env с реальной модельной матрицей или
явно добавить `ENABLE_FUNCTIONS=true` рядом с `PLUGINS`.

### 17. `chat_modes.yml` ссылается на plugins, которых нет в sample allow-list

`chat_modes.yml` многократно ссылается на `web_research`, а также на
`agent_tools` (`bot/chat_modes.yml:104`, `bot/chat_modes.yml:1586`).
В `.env.example:65` этих plugins в `PLUGINS` нет. Registry только логирует
missing tools (`bot/chat_modes_registry.py:67-75`), а runtime-фильтр удаляет
missing plugins из списка (`bot/plugin_manager.py:598-609`).

Почему это баг: режим может обещать tools в prompt/config, но фактически они
не попадут в запрос к модели.

Что сделать: либо добавить plugins в sample allow-list, либо сделать strict
startup validation для production-конфигураций.

### 18. `mcp_server` включён в sample env, но dependency отсутствует в requirements

`bot/plugins/mcp_server.py` импортирует `mcp` (`bot/plugins/mcp_server.py:11-12`),
`.env.example` включает `mcp_server` (`.env.example:65`), но `requirements.txt`
заканчивается без пакета `mcp` (`requirements.txt:41-60`). Import failure при
загрузке plugin логируется и проглатывается (`bot/plugin_manager.py:159-172`).

Почему это баг: plugin заявлен в конфиге, но в стандартной установке может не
загрузиться, а пользователь увидит только log message.

Что сделать: добавить dependency или убрать plugin из sample env; в strict mode
падать при failed-to-load enabled plugin.

### 19. `SKILLS_DIR` в `.env.example` указывает на другой absolute path

В sample env указано `SKILLS_DIR=/srv/chatgpt-telegram-bot/bot/skills`
(`.env.example:66`), тогда как рабочий проект расположен в
`/srv/git_projects/chatgpt-telegram-bot`.

Почему это баг: копирование env создаёт/сканирует не тот каталог skills.

Что сделать: заменить absolute path на относительный или на актуальный путь
проекта.

## LOW

### 20. DB fixture сбрасывает singleton без закрытия thread-local connection

В `tests/test_database.py` fixture напрямую делает `Database._instance = None`
(`tests/test_database.py:19-24`). В самом классе уже есть helper
`Database._reset_singleton()`, который закрывает thread-local connection и
пересоздаёт thread-local storage (`bot/database.py:43-59`).

Почему это проблема: тесты могут оставлять старые connections и маскировать
дефекты жизненного цикла singleton.

Что сделать: использовать `Database._reset_singleton()` в fixture.

### 21. Тест на hardcoded plugin refs не ловит qualified tool names

`tests/test_no_hardcoded_plugin_refs.py` считает только точные quoted occurrences
plugin id (`tests/test_no_hardcoded_plugin_refs.py:58-60`). В core при этом есть
qualified references вроде `agent_tools.deliver_to_user`
(`bot/openai_tool_handler.py:72`, `bot/openai_tool_handler.py:274`).

Почему это проблема: тест может пропустить новые hardcoded references в формате
`plugin.tool`.

Что сделать: расширить regex на qualified names и явно оформить существующие
разрешённые исключения.

## Удалено или объединено из предыдущей версии

- Удалены пункты вне текущего скоупа аудита.
- Дублирующиеся session-race пункты объединены в HIGH 2-4.
- Неподтверждённые или by-design пункты не перенесены: default mutator identity,
  `collect_objects` через `contribute_prompt_fragment`, DDL как plugin-owned
  startup-код, а также общие замечания про размер файлов без конкретного
  дефекта.

## Рекомендуемый порядок исправления

1. HIGH 1-4: tool-call аргументы и инварианты сессий. Это наиболее вероятные
   пользовательские поломки и потери состояния.
2. HIGH 5-6: блокирующий `ask_sync()` и image document в vision handler.
3. MEDIUM 7-15: request-context/tool pipeline, streaming empty responses,
   callback races и plugin isolation.
4. MEDIUM 16-19 и LOW 20-21: конфигурационный drift и тестовые пробелы.
